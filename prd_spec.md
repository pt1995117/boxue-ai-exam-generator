# 智能出题系统 - 产品需求文档 (PRD)

## 1. 项目概述

### 1.1 项目背景
为房地产经纪人培训考试系统提供自动化出题能力，替代人工出题，提高出题效率和质量一致性。系统基于 LangGraph 多智能体协同架构，采用"照猫画虎"（Few-Shot Learning）策略，通过分析知识点和母题库，自动生成高质量的考试题目。

### 1.2 核心价值
- **效率提升**：自动化出题，单题生成时间 < 30秒
- **质量保障**：多智能体协同验证，题目准确率 ≥ 95%
- **风格统一**：参考历史母题，确保题目风格一致
- **智能计算**：自动识别计算需求，支持17种房地产专业计算

### 1.3 核心挑战
1. **知识点覆盖广**：涉及法律、金融、房产交易等多领域（1712条知识点）
2. **计算题复杂**：包含贷款、税费等专业计算（17种计算器）
3. **质量要求高**：题目需准确、干扰项需合理、解析需清晰
4. **风格要求统一**：需与现有408道母题风格保持一致
5. **约束条件多**：题型、难度、模式、掌握程度等多维度约束

## 2. 功能需求

### 2.1 核心功能

#### F1: 智能出题 ✅
- **FR1.1**: 支持从知识库选择章节生成题目
  - **章节范围选择**：
    - 全部章节（默认）
    - 自定义章节（支持多选）
    - 仅计算类章节（自动筛选，关键词：计算、税费、贷款、建筑指标、面积）
  - **掌握程度筛选**：
    - 全部掌握程度（默认）
    - 自定义掌握程度（支持多选）
    - 显示选中范围的知识点数量和掌握程度分布
  
- **FR1.2**: 支持批量生成 (1-200 题)
  - **出题范围模式**：
    - 每个知识点各出一题模式（默认）：自动匹配知识点数量，题型/难度/模式采用默认值（题型单选题，难度随机，模式灵活）
    - 自定义模式：用户指定数量（1-200题），可自定义题型、难度、模式

- **FR1.3**: 支持题型选择 ✅
  - 单选题（4个选项，唯一答案）
  - 多选题（至少4个选项，多个答案）
  - 判断题（固定"正确"/"错误"两个选项）
  - 随机题型（系统自动选择）
  - **题型结构与格式统一规范**：
    - **判断题**：题干必须以判断题结尾，选项固定为“正确/错误”，答案只能是 A 或 B
    - **单选题**：4个选项且仅1个正确答案，答案为单个字母
    - **多选题**：至少4个选项且至少2个正确答案，答案为多个字母组合
    - **括号规范（判断/选择题统一）**：
      - **中文括号**：统一使用“（ ）”
      - **括号前后无空格**
      - **括号内有空格**（示例：`（ ）`）

- **FR1.4**: 支持难度偏好设置 ✅
  - 简单 (0.3-0.5)：直接考察知识点定义、基础概念
  - 中等 (0.5-0.7)：需要理解知识点含义并应用到场景
  - 困难 (0.7-0.9)：需要综合多个知识点、复杂计算或多步推理
  - **随机难度均分策略**：系统自动选择难度，确保低中高难度题目比例均衡（1:1:1）
    - 为每道题随机分配难度范围（简单/中等/困难之一）
    - 使用轮询或随机策略确保整体比例接近 33.3% : 33.3% : 33.3%
  - **约束验证**：所有节点（生成、修复、验证）都严格遵循难度范围

- **FR1.5**: 支持出题筛选条件选择（替代“灵活/严谨”）✅
  - **基础概念/理解记忆**
    - 目标：考察切片知识点本身（定义、规则、条件、结构）
    - 不强制业务场景，可直接围绕知识点命题
    - 仍需忠实原文、答案唯一、解析完整
  - **实战应用/推演**
    - 目标：考察经纪业务中的应用与推演能力
    - 必须关联业务场景（客户咨询、交易流程、合规操作、税费/贷款决策等）
    - 题干和解析需体现“条件 -> 推理 -> 结论”
  - **随机**
    - 系统在“基础概念/理解记忆”和“实战应用/推演”之间自动选择
    - 每题必须严格遵循被选中的筛选条件输出

- **FR1.6**: 生成内容包含完整题目信息
  - 题干（场景化案例题，禁止定义题）
  - 选项（4个选项，维度一致）
  - 正确答案（单选/多选/判断格式）
  - 解析（讲原理、结合情境、口语化）
  - 难度值（0.0-1.0，符合用户指定范围）
  - 考点（知识点完整路径）
  - 知识点层级（一级/二级/三级/四级知识点）
- **人名规范强制校验**：
  - 禁止“姓+女士/先生”“小+姓氏”等称谓
  - Writer 节点必须代码级检测并触发整体改写（显式提示人名违规；不得直接替换为“某某”）
  - Critic 节点代码级校验（只检查不修复）
  - **动态意图约束（仅部分题型触发）**：
    - 仅当题干被识别为 `ENUMERATION`（完整清单/资料包括）或 `EXCLUSION`（不需要/不包括）时，才触发后续强校验
    - 允许**无标签题**，不触发动态约束
    - 禁止 “SELECTION（单点选择/最XX/关键）” 类题型；Writer 与 Critic 均应规避
  - **语义覆盖与专有名词保护**：
    - ENUMERATION/EXCLUSION 的校验应允许**语义等价**的包含项，不要求逐字一致
    - 但专有名词库中的词**禁止同义改写/语义等价替换**，必须原文一致
    - 专有名词命中条件：**关键词召回命中 + 语义语境判定为同一概念**（双条件同时满足）
    - 专有名词库主文件：`房地产行业专有名词新.xlsx`（可预处理缓存为 `教材提取专有名词.txt`）
  - **扩展切片上下文（审题/修复一致）**：
    - 基于当前知识片段路径，补充上一级路径下的**全部切片**
    - 若存在参考母题，基于母题题干+解析检索**相关切片**作为补充
    - Writer/Critic/Fixer 必须基于【当前切片 + 上一级切片全集 + 相似切片】判断严谨性与场景常见性
  - **切片原文透传**：
    - 切片内容在题库与导出时必须**原样保留**
    - 输出中需包含“切片原文/结构化内容”，用于溯源与复核
  - **题型结构与格式规范（必须）**：
    - **判断题**：题干为“描述条件 + 观点结论（体现考点）+ 句号 + 括号（中间空一格）”；选项固定为“正确/错误”；答案为 A/B。
    - **单选题**：题干为“描述条件 + 问题（考点/陈述句）+ 括号（中间空一格）+ 句号”；4 个选项且只有 1 个正确；答案为单字母。
    - **多选题**：题干为“描述条件 + 问题（考点/陈述句）+ 括号（中间空一格）+ 句号”；至少 4 个选项且至少 2 个正确；答案为多字母组合。
    - **括号格式统一**：题干中的括号使用**中文全角括号**，括号前后**不留空格**，括号内**必须有一个空格**，固定为“（ ）”；禁止出现“( ) / () / （）/ （  ）”等变体。

#### F2: 照猫画虎 (Few-Shot Learning) ✅
- **FR2.1**: 根据知识点自动匹配历史相似题目
  - 优先使用知识点映射文件（knowledge_question_mapping.json）
  - 映射文件包含408道母题 → 200个知识点路径的映射
  - 无映射则不返回母题

- **FR2.1.1**: 母题与知识切片高精度关联技术方案 ✅
  - **核心目标**：
    - **自动关联率**：目标实现80%以上的自动关联率
    - **映射准确率**：通过路径硬对齐锁死上下文，确保高准确率
    - **多维输出**：支持"一对多"映射（1个母题可对应多个原子切片）
  
  - **预处理：标准化"脱水" (Normalization)**：
    在进行匹配前，系统必须对母题和知识库的路径进行标准化处理，消除格式噪音：
    - **关键词清洗**：移除"第X篇"、"第X章"、"第X节"、"（了解/掌握/熟悉）"、"-无需修改"等描述性词汇
    - **符号归一化**：将所有全角标点、空格、斜杠统一转换为标准分隔符 `/`
    - **同义词映射**：建立字典，如 `个税` ↔ `个人所得税`，`贝壳` ↔ `贝壳找房/BEIKE`
    - **切片标题前缀剥离**：在进行路径策略操作**之前**，对 KB 切片的**切片标题**（完整路径的最后一节）做前缀剥离，移除「一、」「二、」「（一）」「1、」等结构性字符；剥离后的标题用于**知识端 GPS 路径（kb_gps）**最后一节的构建，以便全路径包含正确命中（例如考点「贝壳战略」可命中切片「二、贝壳战略」）
  
  - **五级阶梯关联策略**：
    
    **策略 1：反向索引复用 (Reverse Index - P0)**
    - **逻辑**：直接碰撞现有的 `knowledge_question_mapping.json`
    - **置信度**：1.0
    - **用途**：保护存量数据，避免重复计算
    - **优先级**：最高优先级，如果匹配到则立即返回
    
    **策略 2：GPS 路径坐标对齐 (Path-Based GPS Match - P1)**
    - **母题端（GPS 坐标）**：`标准化(篇/章/节/考点)`
      - 示例：`交易服务/不动产交易税费/个人所得税/个税计算`
    - **知识端（目标路径）**：`标准化(完整路径)`
    - **匹配规则**：
      - **全路径包含**：仅当双方路径深度均 ≥3（至少含章/节/考点）时，母题的"篇+章+节+考点"完全包含在 KB 路径中才视为命中；置信度 0.95，**立即返回**，不执行后续策略（**不做 BGE 细化**）。用于避免“篇级/章级/节级”总览切片误命中具体题目
      - **输出 method**：全路径记为 `GPS_FullPath`。
    
    **策略 3：法条与编码硬碰撞 (Statute Collision - P2)**
    - **逻辑**：正则表达式提取题干或解析中的法条编号（如《民法典》第215条）
    - **匹配**：若知识切片标题或正文中出现相同法条编号，基础置信度 0.88；**对该批候选母题执行 BGE 细化**：`confidence = 0.88 + 0.07 * (bge_score - 0.5)`，上限 0.95，按「只关联最相关的」保留
    - **适用场景**：法律条文相关的题目和知识切片
    
    **策略 4：语义向量检索 (BGE Vector Retrieval - P3)**
    - **模型**：`BAAI/bge-small-zh-v1.5`
    - **输入构建（Context Buffer）**：
      - **母题输入** = `[标准化路径]` + `[题干]` + `[选项拼合]` + `[解析]`
      - **知识输入** = `[完整路径]` + `[核心内容]`（全文，不做摘要截断）
    - **评分逻辑**：
      - `Score > 0.75`：自动通过，置信度 = Score
      - `0.5 < Score <= 0.75`：进入策略 5（逻辑复核），置信度 = Score
      - `Score <= 0.5`：不匹配
    
    **策略 5：LLM 专家逻辑重排序 (LLM Reranking - P4)**
    - **触发条件**：
      - 常规：策略 4 筛选出的前 3-5 个候选切片（Score > 0.5 且 <= 0.75）
      - **兜底（强制）**：只要策略 1-4 **均未命中**，必须触发 LLM 兜底；取 BGE 排名 Top 3-5 的候选切片（即使 Score <= 0.5）交给 LLM 判断
    - **Prompt 逻辑**：
      > "你是一个房产交易专家。这道题考的是【母题 GPS 路径】，题干是【内容】。请从以下 3 个知识切片中选出能支撑解题的项。如果不相关，请输出 False。"
    - **输出格式**：JSON，包含 `is_related` (bool)、`related_indices` (array)。**LLM 调用强制 JSON 输出**（`response_format: {"type": "json_object"}`）。
    - **置信度**：LLM 判定相关时，用其 BGE 分数做细化：`confidence = 0.80 + 0.10 * (bge_score - 0.5)`，上限 0.90
    - **解析失败处理**：**禁止静默忽略**。若 LLM 返回的 JSON 解析失败（如 Unterminated string、截断等），须将原始响应写入调试文件（如 `llm_rerank_debug_<timestamp>.txt`），然后**抛错终止**，把问题暴露给调用方；不允许 catch 后 `return []` 继续跑。
  
  - **关联原则：只关联最相关的**：
    - **核心原则**：只关联最相关的母题，不关联所有相关的母题
    - **最相关判断标准**：
      - 按策略优先级：策略1 > 策略2 > 策略3 > 策略4 > 策略5
      - 同策略内按置信度降序：置信度高的优先
      - 如果存在多个最相关的母题（置信度相同或接近），可以关联多个
    - **关联阈值**：
      - 策略1（反向索引）：置信度1.0，匹配到立即返回，不再执行后续策略
      - 策略2（GPS路径）：**全路径**匹配到立即返回
      - 策略3（法条碰撞）：经 BGE 细化后只保留置信度最高的匹配（如果多个相同置信度，保留所有）
      - 策略4（BGE向量）：Score > 0.75自动通过，只保留Score最高的前N个（N≤3，如果Score相同或接近，保留所有）
      - 策略5（LLM重排序）：用 BGE 细化置信度，只保留LLM判定为相关的匹配
    - **多匹配处理**：
      - 如果最相关的有多个（置信度相同或接近，差异<0.05），可以关联多个
      - 如果最相关的只有一个，只关联一个
      - 输出时按置信度降序排列
    - **每道题维度过滤**（输出前全局后处理）：
      - **规则**：每道母题只保留置信度最高的切片关联；若多个切片与该题的置信度相同（一致），可保留多个
      - **含义**：同一母题可能被多个切片匹配，输出时仅保留该题下置信度最高的切片；置信度相同时这些切片均保留
      - **实施**：在写入 `knowledge_question_mapping.json` 前，对完整映射做按题过滤，再写盘
  
  - **针对"1:N 映射"的特殊处理逻辑**（已废弃，改为"只关联最相关的"）：
    - ~~**父节点聚合**：如果策略 2 匹配到了某个"节"层级，系统会自动将该"节"下所有的原子切片（Slice）列为候选~~（已废弃）
    - ~~**置信度衰减**：在 JSON 输出中，首位映射置信度最高，随后的关联项按相关度递减~~（已废弃）
  
  - **批量映射执行方式（按母题遍历）**：
    - **遍历顺序**：对每道母题，在全量知识切片中查找匹配的切片；即「一道题 × 全量切片」。输出仍为 `knowledge_question_mapping.json`（slice → matched_questions）。
    - **BGE 优化**：全量切片 embedding **预计算一次**；每道母题 embedding 计算一次，与所有切片向量做相似度。母题数远小于切片数时，可显著减少 BGE 调用（如 10 题 × 536 切片：536 + 10 次 embedding，替代按切片遍历时的 536×(1+10) 次）。
  
  - **输出格式**：`knowledge_question_mapping.json`
    ```json
    {
      "0": {
        "完整路径": "第一篇 > 第一章 > ...",
        "掌握程度": "掌握",
        "matched_questions": [
          {
            "question_index": 3,
            "confidence": 0.95,
            "method": "GPS_FullPath",
            "evidence": {
              "reason": "路径完全匹配：第四篇/第四章/第二节/个人所得税计算"
            }
          },
          {
            "question_index": 5,
            "confidence": 0.85,
            "method": "LLM_Logic",
            "evidence": {
              "reason": "逻辑支撑：包含计算个税所需的扣除项标准"
            }
          }
        ],
        "total_matches": 2,
        "methods_used": ["GPS_FullPath", "LLM_Logic"]
      }
    }
    ```
  
  - **实施步骤**：
    1. **第一步**：生成"干净"的映射候选 (Batch Process) - **按母题遍历**，每道题 × 全量切片，执行策略 1-3，解决 50% 以上的简单匹配
    2. **第二步**：执行语义召回与排序 (Vector & LLM) - BGE 全量切片预计算；对复杂题型执行向量检索和 LLM 复核
    3. **第三步**：输出关联报告 (Validation Report) - 生成 CSV，包含：`母题题干 | 考点路径 | 关联 Slice ID | 关联理由 | 置信度`

- **FR2.2**: 根据题型过滤母题 ✅
  - 单选题配单选题母题
  - 多选题配多选题母题
  - 判断题配判断题母题（选项1=正确，选项2=错误）
  - **随机题型优先策略（新增）**：
    - 当出题配置为“随机”且当前切片存在已确认的母题映射时，系统应优先从该切片已关联母题的题型集合中选题型。
    - 例如：某切片关联母题包含“判断题 + 单选题”，则该切片后续出题应优先生成判断题或单选题，而非无约束随机。
    - 若该切片无已确认映射或无法识别母题题型，才回退到 Router 推荐题型。

  - **时间继承**：如果原文未给出具体时间，题干与解析不得添加具体年份/日期；仅保留相对时间（如"满5年"）
  - **时间使用规则**：若原文明示时间，才可使用对应年份/日期；判定逻辑必须严格遵循教材规则（如"满5年"的计算）
  - **时间干扰项**：允许设计关于时间的干扰项（如设置一个时间未满的情景作为错误选项），但解析必须清晰指出不符合哪条时间规则
  - **计算时间精度**：时间/日期题必须用 datetime 精确到天，禁止用年份直接相减

#### F6: 反馈循环机制 ✅
- **FR6.1**: Fixer → Critic 循环
  - 修复后自动回到验证
  - 最多3次修复循环

- **FR6.2**: Critic → Router 重路由
  - 严重问题触发重新路由
  - Router检测重路由并清理旧状态

- **FR6.3**: 状态清理
  - Router检测重路由并清理旧状态
  - 保留必要的上下文（prev_final_json, prev_critic_feedback）

- **FR6.4**: retry_count 管理
  - 最多3次重试，避免无限循环
  - 超限后触发自愈机制（bypass Router直接重新生成）
  - 自愈机制：当retry_count >= 3时，直接调用Fixer重新生成，不再经过Router

- **FR6.5**: 智能决策机制 ✅
  - **通过**：critic_result.passed = True → 结束流程
  - **轻微问题**：issue_type = 'minor' → Fixer修复 → 回到Critic验证
  - **严重问题**：issue_type = 'major' → 
    - retry_count < 2：Router重新路由
    - retry_count >= 2：Fixer强制修复
  - **超限自愈**：retry_count >= 3 → 自愈机制（直接Fixer重新生成，不再经过Router）

    - `一级知识点`、`二级知识点`、`三级知识点`、`四级知识点`: 从`完整路径`按`>`分割后提取
    - `来源路径`: 在`app.py`中设置，用于展示和导出
    - `_was_fixed`: 布尔值，标记是否经过Fixer修复
    - `是否修复`: 字符串（"是"/"否"），用于UI展示
  - **数据验证**: 使用数据模型进行验证，捕获格式错误

## 5. 约束条件

### 5.1 技术约束
- **C1**: 必须使用 Streamlit 作为 UI 框架 ✅

- **C2**: 必须使用 LangGraph 进行智能体编排 ✅

- **C3**: LLM 需支持 JSON 格式输出 ✅
  - 所有节点都要求LLM返回JSON格式
  - 支持从markdown代码块中提取JSON

- **C4**: Python 环境 ✅
  - 需要安装相关依赖包（pandas、openpyxl、pydantic、scikit-learn、openai、streamlit、langgraph、langchain等）

### 5.2 业务约束
- **C5**: 题目内容必须 100% 准确，不得出现幻觉
- **C6**: 题干中禁止出现"根据材料"、"依据参考资料"等提示语
- **C7**: 干扰项必须似是而非，不能一眼假
- **C8**: 必须出场景化案例题，禁止定义题
- **C9**: 选项维度必须一致，禁止跨维度
- **C10**: 严禁直接照搬原文案例中的具体数据（必须做数据重构）
- **C11**: 难度值必须符合用户指定的难度范围
- **C12**: 题型必须符合用户选择（单选/多选/判断）
- **C13**: 出题筛选条件必须符合用户选择（基础概念/理解记忆、实战应用/推演、随机）
- **C14**: 地理一致性：教材限定城市时，题干必须在该城市；通用规则不得写具体城市
- **C15**: 时间逻辑：原文未给具体时间时，不得添加具体年份/日期；仅保留相对时间
- **C16**: 禁止题干直接给出答案：题干中不得直接包含正确答案的关键词，导致考生无需理解即可通过文字匹配选出答案。但允许正确答案选项与教材原文定义一致。
- **C17**: 禁止生造词：必须使用标准业务术语，禁止AI幻觉词汇
- **C18**: 禁止"最XX"考法：禁止用"最重要/最关键/重点/主要"等表述
- **C19**: 唯一答案强制校验：逐条假设每个错误选项为真，验证是否"必错"

### 5.3 数据约束
- **C20**: 知识库和母题库为静态数据，不可修改 ✅
  - 知识库和母题库文件为只读，系统仅读取，不写入

- **C21**: 母题数据可能存在 NaN，需过滤 ✅
  - 自动过滤NaN值和空字符串
  - 必需字段检查：题干、选项1、选项2、正确答案、解析
  - 无效数据不参与母题检索

- **C22**: 知识点映射文件需预先生成 ✅
  - 如果映射文件不存在，不返回母题（不做语义回退）

- **C23**: 知识库字段约束 ✅
  - **必需字段**：`完整路径`、`核心内容`（如果缺失，从`结构化内容`自动构建）
  - **可选字段**：`掌握程度`、`结构化内容`、`Bot专用切片`
  - **结构化内容字段**：`context_before`、`context_after`、`tables`、`formulas`、`examples`、`key_params`（可选）

- **C24**: 母题库字段约束 ✅
  - **必需字段**（用于数据质量过滤）：`题干`、`选项1`、`选项2`、`正确答案`、`解析`
  - **可选字段**：`选项3-8`、`考点`、`难度值`
  - **题型识别**：
    - 判断题：选项1为"正确"且选项2为"错误"
    - 多选题：正确答案长度>1且所有字符在A-E中
    - 单选题：默认

- **C25**: 输出题目字段约束 ✅
  - **必需字段**（`ExamQuestion`模型）：`题干`、`选项1`、`选项2`、`正确答案`、`解析`、`难度值`
  - **可选字段**：`选项3-8`（默认空字符串）
  - **正确答案格式**：正则表达式`^[ABCDEFGH]+$`（A-H或组合）
  - **难度值范围**：0.0-1.0（`ge=0, le=1`）
  - **自动生成字段**：`考点`、`一级知识点`、`二级知识点`、`三级知识点`、`四级知识点`、`来源路径`、`_was_fixed`、`是否修复`

## 8. 用户场景

### 场景 1: 生成中等难度的单选题
1. 用户打开系统
2. 配置 API Key（DeepSeek）
3. 选择章节 "第一篇 > 第一章 > 第一节"
4. 选择题型 "单选题"
5. 选择难度 "中等 (0.5-0.7)"
6. 选择模式 "灵活"
7. 设置数量 "5 题"
8. 点击"开始出题"
9. 系统展示：
   - Router 决策（知识点、掌握程度、派发专家）
   - 母题范例（参考的3道母题）
   - 生成过程（初稿、格式化、验证）
   - 最终题目（难度值在0.5-0.7范围内）
10. 用户复制题目

### 场景 2: 生成计算题（基础概念/理解记忆）
1. 用户选择章节 "第二篇 (金融税费相关)"
2. 勾选"仅选中计算类章节"
3. 选择题型 "单选题"
4. 选择难度 "困难 (0.7-0.9)"
5. 选择模式 "严谨"
6. 点击"开始出题"
7. 系统：
   - Router 识别为计算类 → CalculatorAgent → calculator_node
   - 获取计算类母题范例
   - Calculator 动态生成Python计算代码并执行（展示函数、参数、结果）
   - 生成包含计算的题目（基础概念/理解记忆，可不做场景化包装）
   - Writer 格式化（验证难度值在0.7-0.9范围内）
   - Critic 用计算器验证答案
   - 输出最终题目（难度值符合要求）
8. 用户查看题目和计算详情

### 场景 3: 生成判断题（简单难度）
1. 用户选择章节
2. 选择题型 "判断题"
3. 选择难度 "简单 (0.3-0.5)"
4. 选择模式 "灵活"
5. 点击"开始出题"
6. 系统：
   - 只检索判断题母题（选项1=正确, 选项2=错误）
   - 生成只有"正确/错误"两个选项的题目
   - 难度值在0.3-0.5范围内
7. 输出判断题

### 场景 4: 修复不符合难度要求的题目
1. 用户选择难度 "中等 (0.5-0.7)"
2. 系统生成题目，但难度值为0.3（不符合要求）
3. Critic 检测到难度不符合要求，标记为严重问题
4. 触发 Router 重新路由
5. 重新生成题目，难度值调整为0.6（符合要求）
6. 输出最终题目

## 9. 风险与应对

### 风险 1: LLM 生成不稳定
- **应对**: Critic 验证 + Fixer 修复 + 最多 3 次重试 + 自愈机制

### 风险 2: 母题数量不足
- **应对**: 仅使用知识点映射文件；无映射则不返回母题

### 风险 3: 计算器调用失败
- **应对**: 错误捕获 + 日志记录 + 降级为概念题

### 风险 4: 数据质量问题
- **应对**: 数据验证 + NaN 过滤 + 必需字段检查

### 风险 5: 难度值不符合要求
- **应对**: ✅ 多节点验证（Writer、Critic、Fixer）+ 自动调整机制

### 风险 6: 题型不符合要求
- **应对**: ✅ 所有节点严格遵循题型约束 + 修复模式也遵循

### 风险 7: 模式不符合要求
- **应对**: ✅ 提示词中明确区分三档筛选条件 + 修复模式也遵循

### 风险 8: 代码执行安全问题
- **应对**: ✅ 沙箱执行环境 + 模块白名单 + 超时控制 + 内置函数限制

### 风险 9: 地理/时间逻辑错误
- **应对**: ✅ Writer节点检查 + Critic节点验证 + 明确Fail条件

### 风险 10: 同义反复/信息泄露
- **应对**: ✅ Critic节点检测题干与正确选项文字重合度（>60%即Fail）

### 风险 11: AI幻觉/生造词
- **应对**: ✅ Critic节点检测生造词（如"外接"、"上交"等非标准术语）
