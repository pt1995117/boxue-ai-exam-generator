# 智能出题系统 - 产品需求文档 (PRD)

## 1. 项目概述

### 1.1 项目背景
为房地产经纪人培训考试系统提供自动化出题能力，替代人工出题，提高出题效率和质量一致性。系统基于 LangGraph 多智能体协同架构，采用"照猫画虎"（Few-Shot Learning）策略，通过分析知识点和母题库，自动生成高质量的考试题目。

### 1.2 核心价值
- **效率提升**：自动化出题，单题生成时间 < 30秒
- **质量保障**：多智能体协同验证，题目准确率 ≥ 95%
- **风格统一**：参考历史母题，确保题目风格一致
- **智能计算**：自动识别计算需求，支持16种房地产专业计算（当前实现）
- **对经纪人工作有帮助（必达）**：题目的核心标准是**对经纪人的工作是否有帮助**，不是为背书。可以是实操类（业务判断、执行流程、服务客户、规避风险），也可以是理解规则、合规要点、公司文化等——只要对工作有正向作用即可。**禁止**的是对工作无帮助的纯背书题：（1）仅考「概念归类/概念辨析」（如定义 vs 目的 vs 方式）、对实际作业无指导的题目；（2）仅考「教材把哪一条称为核心/主要/关键」、实务上多选项都重要、区分只靠记教材措辞的刁钻题。若 Specialist 判断当前知识点无法形成对经纪人工作有帮助的题目，可拒绝出题并直接结束。

### 1.3 核心挑战
1. **知识点覆盖广**：涉及法律、金融、房产交易等多领域（1712条知识点）
2. **计算题复杂**：包含贷款、税费等专业计算（16种计算器，当前实现）
3. **质量要求高**：题目需准确、干扰项需合理、解析需清晰
4. **风格要求统一**：需与现有408道母题风格保持一致
5. **约束条件多**：题型、难度、模式、掌握程度等多维度约束

## 2. 功能需求

### 2.1 核心功能

#### F1: 智能出题 ✅
- **FR1.1**: 支持从知识库选择章节生成题目
  - **章节范围选择**：
    - 全部章节（默认）
    - 自定义章节（支持多选）
    - 仅计算类章节（自动筛选，关键词：计算、税费、贷款、建筑指标、面积）
  - **掌握程度筛选**：
    - 全部掌握程度（默认）
    - 自定义掌握程度（支持多选）
    - 显示选中范围的知识点数量和掌握程度分布
  
- **FR1.2**: 支持批量生成 (1-200 题)
  - **出题范围模式**：
    - 每个知识点各出一题模式（默认）：自动匹配知识点数量，题型/难度/模式采用默认值（题型单选题，难度随机，模式灵活）
    - 自定义模式：用户指定数量（1-200题），可自定义题型、难度、模式

- **FR1.3**: 支持题型选择 ✅
  - 单选题（4个选项，唯一答案）
  - 多选题（至少4个选项，多个答案）
  - 判断题（固定"正确"/"错误"两个选项）
  - 随机题型（系统自动选择）
  - **题型结构与格式统一规范**：
    - **判断题**：题干必须以判断题结尾，选项固定为“正确/错误”，答案只能是 A 或 B
    - **单选题**：4个选项且仅1个正确答案，答案为单个字母
    - **多选题**：至少4个选项且至少2个正确答案，答案为多个字母组合
    - **括号规范（判断/选择题统一）**：
      - **中文括号**：统一使用“（ ）”
      - **括号前后无空格**
      - **括号内为全角空格**（示例：`（　）`，U+3000）

- **FR1.4**: 支持难度偏好设置 ✅
  - 简单 (0.3-0.5)：直接考察知识点定义、基础概念
  - 中等 (0.5-0.7)：需要理解知识点含义并应用到场景
  - 困难 (0.7-0.9)：需要综合多个知识点、复杂计算或多步推理
  - **随机难度均分策略**：系统自动选择难度，确保低中高难度题目比例均衡（1:1:1）
    - 为每道题随机分配难度范围（简单/中等/困难之一）
    - 使用轮询或随机策略确保整体比例接近 33.3% : 33.3% : 33.3%
  - **约束验证**：所有节点（生成、修复、验证）都严格遵循难度范围

- **FR1.5**: 支持出题筛选条件选择（替代“灵活/严谨”）✅
  - **基础概念/理解记忆**
    - 目标：考察切片知识点本身（定义、规则、条件、结构）
    - 不强制业务场景，可直接围绕知识点命题
    - 仍需忠实原文、答案唯一、解析完整
  - **实战应用/推演**
    - 目标：考察经纪业务中的应用与推演能力
    - 必须关联业务场景（客户咨询、交易流程、合规操作、税费/贷款决策等）
    - 题干和解析需体现“条件 -> 推理 -> 结论”
  - **随机**
    - 系统在“基础概念/理解记忆”和“实战应用/推演”之间自动选择
    - 每题必须严格遵循被选中的筛选条件输出
  - **当前代码口径说明**：
    - Streamlit 出题页当前仍展示“灵活/严谨”选项
    - 引擎侧会将“灵活/严谨”映射为内部筛选语义（兼容旧配置）

- **FR1.6**: 生成内容包含完整题目信息
  - 题干（场景化案例题，禁止定义题）
  - 选项（4个选项，维度一致）
  - 正确答案（单选/多选/判断格式）
  - 解析（讲原理、结合情境、口语化）
  - 难度值（0.0-1.0，符合用户指定范围）
  - 考点（知识点完整路径）
  - 知识点层级（一级/二级/三级/四级知识点）
- **FR1.7**: Specialist 可拒绝出题与公司文化例外 ✅
  - **公司文化类**：知识点属公司文化（企业文化、价值观、品牌理念等）时，允许出理解/记忆类题目，不强制场景化实操题；Critic 对该类题目不适用「仅考概念辨析/仅考教材核心·主要·关键」的 Fail。
  - **拒绝出题**：Specialist 若判断当前知识点无法形成对经纪人工作有帮助的题目（如内容无法支撑对工作有指导意义的考法），可拒绝出题并直接结束该题，不进入 Writer/Critic；图中 specialist 后条件边：refuse_to_generate 则 END，否则 writer。
- **人名规范强制校验**：
  - 禁止“姓+女士/先生”“小+姓氏”等称谓
  - Writer 节点必须代码级检测并触发整体改写（显式提示人名违规；不得直接替换为“某某”）
  - Critic 节点代码级校验（只检查不修复）
  - **动态意图约束（仅部分题型触发）**：
    - 仅当题干被识别为 `ENUMERATION`（完整清单/资料包括）或 `EXCLUSION`（不需要/不包括）时，才触发后续强校验
    - 允许**无标签题**，不触发动态约束
    - 禁止 “SELECTION（单点选择/最XX/关键）” 类题型；Writer 与 Critic 均应规避
  - **语义覆盖与专有名词保护**：
    - ENUMERATION/EXCLUSION 的校验应允许**语义等价**的包含项，不要求逐字一致
    - 但专有名词库中的词**禁止同义改写/语义等价替换**，必须原文一致
    - 专有名词命中条件：**关键词召回命中 + 语义语境判定为同一概念**（双条件同时满足）
    - 专有名词库主文件：`房地产行业专有名词新.xlsx`（可预处理缓存为 `教材提取专有名词.txt`）
  - **扩展切片上下文（审题/修复一致）**：
    - 基于当前知识片段路径，补充上一级路径下的**全部切片**
    - 若存在参考母题，基于母题题干+解析检索**相关切片**作为补充
    - Writer/Critic/Fixer 必须基于【当前切片 + 上一级切片全集 + 相似切片】判断严谨性与场景常见性
  - **切片原文透传**：
    - 切片内容在题库与导出时必须**原样保留**
    - 输出中需包含“切片原文/结构化内容”，用于溯源与复核
  - **题型结构与格式规范（必须）**：
    - **判断题**：题干为“描述条件 + 观点结论（体现考点）+ 句号 + 括号（中间空一格）”；选项固定为“正确/错误”；答案为 A/B。
    - **单选题**：题干为“描述条件 + 问题（考点/陈述句）+ 括号（中间空一格）+ 句号”；4 个选项且只有 1 个正确；答案为单字母。
    - **多选题**：题干为“描述条件 + 问题（考点/陈述句）+ 括号（中间空一格）+ 句号”；至少 4 个选项且至少 2 个正确；答案为多字母组合。
    - **括号格式统一**：题干中的括号使用**中文全角括号**，括号前后**不留空格**，括号内**必须为一个全角空格**（U+3000），固定为“（　）”；禁止出现“( ) / () / （ ）/ （  ）”等变体。
  - **选项单位（必须写入提示词）**：选项中有单位时，**必须**将单位提到题干中，**不得**在选项中反复出现单位。单位（如元、万元、平方米、年、%）应写在题干设问处（如「……额度为（　）万元」），选项只写数值（如 6、8、10、12）。
  - **题干设问表述规范（含前提，必须写入提示词）**：
    - **前提**：**仅当题干意思/意图为“判断表述正确与否”或“判断XX做法正确与否”时**，才强制采用下列表述标准；**题干意思中没有这些表述的**（如纯计算题、纯概念选择题、定义类判断题等），**不强制要求**，不得因未使用下列模板而判为不合格。
    - **判断题**：当题干意思为“做法/表述正确或错误”时，**严格**使用「XX做法正确/错误」，**禁止**写成「XX的做法是正确的」；定义类（题干含“属于/是指/定义/概念”等）可不强制该模板。
    - **选择题设问**：当题干意思为“以下表述正确/错误”时——**单选题**用「以下表述正确的是（　）。」，**多选题**用「以下表述正确的有（　）。」或「以下表述正确的包括（　）。」；题干意思非此类时不强制。
  - **题干简练与无关表述（必须写入提示词）**：
    - 题干建议 120 字以内；**题干较长时重点注意**：剔除与本题**毫无关系**的表达，不要让题干变得没必要的复杂、逻辑没必要的绕。
    - 只保留与解题/考点直接相关的信息；去掉对答案没有影响的背景、铺垫与冗余描述。

#### F2: 照猫画虎 (Few-Shot Learning) ✅
- **FR2.1**: 根据知识点自动匹配历史相似题目
  - 优先使用知识点映射文件（knowledge_question_mapping.json）
  - 映射文件包含408道母题 → 200个知识点路径的映射
  - 无映射则不返回母题

- **FR2.1.1**: 母题与知识切片高精度关联技术方案 ✅
  - **核心目标**：
    - **自动关联率**：目标实现80%以上的自动关联率
    - **映射准确率**：通过路径硬对齐锁死上下文，确保高准确率
    - **多维输出**：支持"一对多"映射（1个母题可对应多个原子切片）
  
  - **预处理：标准化"脱水" (Normalization)**：
    在进行匹配前，系统必须对母题和知识库的路径进行标准化处理，消除格式噪音：
    - **关键词清洗**：移除"第X篇"、"第X章"、"第X节"、"（了解/掌握/熟悉）"、"-无需修改"等描述性词汇
    - **符号归一化**：将所有全角标点、空格、斜杠统一转换为标准分隔符 `/`
    - **同义词映射**：建立字典，如 `个税` ↔ `个人所得税`，`贝壳` ↔ `贝壳找房/BEIKE`
    - **切片标题前缀剥离**：在进行路径策略操作**之前**，对 KB 切片的**切片标题**（完整路径的最后一节）做前缀剥离，移除「一、」「二、」「（一）」「1、」等结构性字符；剥离后的标题用于**知识端 GPS 路径（kb_gps）**最后一节的构建，以便全路径包含正确命中（例如考点「贝壳战略」可命中切片「二、贝壳战略」）
  
  - **五级阶梯关联策略**：
    
    **策略 1：反向索引复用 (Reverse Index - P0)**
    - **逻辑**：直接碰撞现有的 `knowledge_question_mapping.json`
    - **置信度**：1.0
    - **用途**：保护存量数据，避免重复计算
    - **优先级**：最高优先级，如果匹配到则立即返回
    
    **策略 2：GPS 路径坐标对齐 (Path-Based GPS Match - P1)**
    - **母题端（GPS 坐标）**：`标准化(篇/章/节/考点)`
      - 示例：`交易服务/不动产交易税费/个人所得税/个税计算`
    - **知识端（目标路径）**：`标准化(完整路径)`
    - **匹配规则**：
      - **全路径包含**：仅当双方路径深度均 ≥3（至少含章/节/考点）时，母题的"篇+章+节+考点"完全包含在 KB 路径中才视为命中；置信度 0.95，**立即返回**，不执行后续策略（**不做 BGE 细化**）。用于避免“篇级/章级/节级”总览切片误命中具体题目
      - **输出 method**：全路径记为 `GPS_FullPath`。
    
    **策略 3：法条与编码硬碰撞 (Statute Collision - P2)**
    - **逻辑**：正则表达式提取题干或解析中的法条编号（如《民法典》第215条）
    - **匹配**：若知识切片标题或正文中出现相同法条编号，基础置信度 0.88；**对该批候选母题执行 BGE 细化**：`confidence = 0.88 + 0.07 * (bge_score - 0.5)`，上限 0.95，按「只关联最相关的」保留
    - **适用场景**：法律条文相关的题目和知识切片
    
    **策略 4：语义向量检索 (BGE Vector Retrieval - P3)**
    - **模型**：`BAAI/bge-small-zh-v1.5`
    - **输入构建（Context Buffer）**：
      - **母题输入** = `[标准化路径]` + `[题干]` + `[选项拼合]` + `[解析]`
      - **知识输入** = `[完整路径]` + `[核心内容]`（全文，不做摘要截断）
    - **评分逻辑**：
      - `Score > 0.75`：自动通过，置信度 = Score
      - `0.5 < Score <= 0.75`：进入策略 5（逻辑复核），置信度 = Score
      - `Score <= 0.5`：不匹配
    
    **策略 5：LLM 专家逻辑重排序 (LLM Reranking - P4)**
    - **触发条件**：
      - 常规：策略 4 筛选出的前 3-5 个候选切片（Score > 0.5 且 <= 0.75）
      - **兜底（强制）**：只要策略 1-4 **均未命中**，必须触发 LLM 兜底；取 BGE 排名 Top 3-5 的候选切片（即使 Score <= 0.5）交给 LLM 判断
    - **Prompt 逻辑**：
      > "你是一个房产交易专家。这道题考的是【母题 GPS 路径】，题干是【内容】。请从以下 3 个知识切片中选出能支撑解题的项。如果不相关，请输出 False。"
    - **输出格式**：JSON，包含 `is_related` (bool)、`related_indices` (array)。**LLM 调用强制 JSON 输出**（`response_format: {"type": "json_object"}`）。
    - **置信度**：LLM 判定相关时，用其 BGE 分数做细化：`confidence = 0.80 + 0.10 * (bge_score - 0.5)`，上限 0.90
    - **解析失败处理**：**禁止静默忽略**。若 LLM 返回的 JSON 解析失败（如 Unterminated string、截断等），须将原始响应写入调试文件（如 `llm_rerank_debug_<timestamp>.txt`），然后**抛错终止**，把问题暴露给调用方；不允许 catch 后 `return []` 继续跑。
  
  - **关联原则：只关联最相关的**：
    - **核心原则**：只关联最相关的母题，不关联所有相关的母题
    - **最相关判断标准**：
      - 按策略优先级：策略1 > 策略2 > 策略3 > 策略4 > 策略5
      - 同策略内按置信度降序：置信度高的优先
      - 如果存在多个最相关的母题（置信度相同或接近），可以关联多个
    - **关联阈值**：
      - 策略1（反向索引）：置信度1.0，匹配到立即返回，不再执行后续策略
      - 策略2（GPS路径）：**全路径**匹配到立即返回
      - 策略3（法条碰撞）：经 BGE 细化后只保留置信度最高的匹配（如果多个相同置信度，保留所有）
      - 策略4（BGE向量）：Score > 0.75自动通过，只保留Score最高的前N个（N≤3，如果Score相同或接近，保留所有）
      - 策略5（LLM重排序）：用 BGE 细化置信度，只保留LLM判定为相关的匹配
    - **多匹配处理**：
      - 如果最相关的有多个（置信度相同或接近，差异<0.05），可以关联多个
      - 如果最相关的只有一个，只关联一个
      - 输出时按置信度降序排列
    - **每道题维度过滤**（输出前全局后处理）：
      - **规则**：每道母题只保留置信度最高的切片关联；若多个切片与该题的置信度相同（一致），可保留多个
      - **含义**：同一母题可能被多个切片匹配，输出时仅保留该题下置信度最高的切片；置信度相同时这些切片均保留
      - **实施**：在写入 `knowledge_question_mapping.json` 前，对完整映射做按题过滤，再写盘
  
  - **针对"1:N 映射"的特殊处理逻辑**（已废弃，改为"只关联最相关的"）：
    - ~~**父节点聚合**：如果策略 2 匹配到了某个"节"层级，系统会自动将该"节"下所有的原子切片（Slice）列为候选~~（已废弃）
    - ~~**置信度衰减**：在 JSON 输出中，首位映射置信度最高，随后的关联项按相关度递减~~（已废弃）
  
  - **批量映射执行方式（按母题遍历）**：
    - **遍历顺序**：对每道母题，在全量知识切片中查找匹配的切片；即「一道题 × 全量切片」。输出仍为 `knowledge_question_mapping.json`（slice → matched_questions）。
    - **BGE 优化**：全量切片 embedding **预计算一次**；每道母题 embedding 计算一次，与所有切片向量做相似度。母题数远小于切片数时，可显著减少 BGE 调用（如 10 题 × 536 切片：536 + 10 次 embedding，替代按切片遍历时的 536×(1+10) 次）。
  
  - **输出格式**：`knowledge_question_mapping.json`
    ```json
    {
      "0": {
        "完整路径": "第一篇 > 第一章 > ...",
        "掌握程度": "掌握",
        "matched_questions": [
          {
            "question_index": 3,
            "confidence": 0.95,
            "method": "GPS_FullPath",
            "evidence": {
              "reason": "路径完全匹配：第四篇/第四章/第二节/个人所得税计算"
            }
          },
          {
            "question_index": 5,
            "confidence": 0.85,
            "method": "LLM_Logic",
            "evidence": {
              "reason": "逻辑支撑：包含计算个税所需的扣除项标准"
            }
          }
        ],
        "total_matches": 2,
        "methods_used": ["GPS_FullPath", "LLM_Logic"]
      }
    }
    ```
  
  - **实施步骤**：
    1. **第一步**：生成"干净"的映射候选 (Batch Process) - **按母题遍历**，每道题 × 全量切片，执行策略 1-3，解决 50% 以上的简单匹配
    2. **第二步**：执行语义召回与排序 (Vector & LLM) - BGE 全量切片预计算；对复杂题型执行向量检索和 LLM 复核
    3. **第三步**：输出关联报告 (Validation Report) - 生成 CSV，包含：`母题题干 | 考点路径 | 关联 Slice ID | 关联理由 | 置信度`

    - **题干设问前提**：题干意思中没有“表述正确/错误”“做法正确/错误”等表述的，不强制设问模板；有此意思的须按规范执行。

    - 解析结构：先摆事实，再引规则，最后结论；可选补充错误选项为什么错
    - 错字修复：发现明显错别字、乱码或奇怪词语，必须直接改正

  - **出题节点一致性自检**：

  - **题型校验策略**：
    - **随机模式**：如果 config 题型为"随机"，Critic **不校验**题型一致性，保留 state 的题型
    - **指定题型模式**：如果 config 题型为具体类型（单选/多选/判断），Critic **必须校验** state 的题型是否与 config 一致，不一致则标记为 **major 问题（failed）**
  - **反向解题验证**：能根据题目条件推导出唯一答案（最高裁决优先级）
    - 在完全忽略生成者声称答案的前提下，仅基于题干条件+教材规则推导
    - Fail条件：无法计算（缺关键数值/条件）、存在多条合理推导路径、需要考生"猜规则"
  - **答案一致性验证**：Critic推导的答案与生成答案一致
  - **信息不对称校验**：检查是否遗漏判定条件、母题冲突检查
  - **计算验证**：对计算题自动调用计算器验证
  - **难度验证**：检查难度值是否在指定范围内
  - **质量检查**：
    - 语境强度（强/中/弱）
    - 选项维度一致性
    - 解析有效性
    - 题干直接给出答案检测（题干中是否直接包含正确答案的关键词，导致无需理解即可选出）
    - AI幻觉/非人话检测（生造词检测）
    - 专有名词锁词审计：`term_locks` 命中词若在题干/选项/解析被改写，直接判定不通过
  - **格式代码校验（Critic）**：
    - 在 Critic 内部先用代码校验题干/选项/答案格式（括号、句号位置、选项末尾标点、答案格式）
    - 将代码校验结果提供给 Critic 大模型作为参考
    - 由大模型在最终 JSON 中汇总输出（reason/quality_issues 体现）
  - **题干与选项组合可读性复核（Critic 侧）**：
    - 适用范围同 Writer：所有使用“题干+占位括号（　）+选项字母”结构的单选/多选/判断题
    - Critic 必须在反向解题与一般质量检查之外，显式执行一轮“选项代入可读性”复核：逐一将每个选项文本代入题干括号位置，形成完整句子
    - 若发现存在 1 个及以上选项在代入后形成的句子读起来明显别扭、语法错误或语义断裂（即便答案本身正确），必须至少作为 **quality_issues** 中的独立问题输出
    - 当可读性问题影响考生理解或可能导致误解考点时，应将该问题归类为 **major**，并通过 Fixer 或重路由要求整体改写；轻微措辞问题可标记为 **minor**
    - 若 Critic 未执行该维度的代入检查或检查逻辑无法覆盖全部选项，视为不符合本 PRD 约束
  - **全量问题输出（Critic）**：
    - Critic 必须继续完成全部检查，即使发现格式问题也不得提前返回
    - Critic 输出需包含完整问题清单（含格式/逻辑/缺失条件/质量/难度/解析/计算代码）
  - **材料清单结构化校验（Critic）**：

    - 若题干问“还需提供/补充提供材料”，且缺失项 > 1 → 直接判定 Fail

  - **地理与范围审计**：检查城市一致性、时间逻辑

  - 若知识切片未提及具体年份，题干/选项/解析中不得出现具体公历年份（如2024年）

  - **逻辑自洽性审计**：比对判定结果而非机械比对数字
  - **选项逻辑审计**：干扰项应考察易错点，而非纯粹随机数字

- **FR3.5**: Fixer 节点错误修复 ✅
  - **题型读取策略**：优先从 state 读取上游节点确定的题型，若无则从 config 读取
  - **修复策略**：
    - fix_explanation：只修改解析
    - fix_question：修改题干/选项/答案
    - fix_both：同时修正题目和解析
    - regenerate：重写题目
  - **约束遵循**：修复时严格遵循题型、模式、难度约束
  - **难度调整**：自动调整难度值到指定范围
  - **格式代码修复（Fixer）**：
    - 在 Fixer 返回前执行代码级格式修复（题干括号/句号位置、选项末尾标点、答案格式、选项数量）
    - 修复结果写回 final_json，并在 UI 展示修复后的完整内容
  - **专有名词锁词修复（Fixer）**：
    - Fixer 修复时禁止改写 `term_locks` 命中术语；如需重写句子，术语必须按原词回填
  - **工作流**：Fixer修复后回到Critic重新验证，形成循环
  - **路由策略**：若经过 Fixer 后 Critic 仍判定为严重问题（major），则进入 reroute

- **FR3.6**: 工作流架构 ✅
  - **节点列表**（6个节点）：
    - router_node：路由决策
    - specialist_node：非计算专家节点
    - calculator_node：计算专家节点
    - writer_node：格式化标准化
    - critic_node：质量验证
    - fixer_node：错误修复
  - **工作流路径**：
    - 入口：router
    - router → calculator/specialist（条件路由）
    - calculator/specialist → writer
    - writer → critic
    - critic → pass/fix/reroute/self_heal（条件路由）
      - pass → END（通过，结束）
      - fix → fixer（轻微问题，修复）
      - reroute → router（严重问题，重新路由）
      - self_heal → END（重试≥3次，自愈输出）
    - fixer → critic（形成循环）
  - **Writer 内部分层工作流（新增）**：
    - `writer` 节点内部按固定顺序执行：`Normalize -> Validate -> Polish`
    - 若 `Validate.passed = true`，可跳过 `Polish` 直接输出
    - 若 `Validate.passed = false`，仅允许 `Polish` 按 issue_code 定向修复后再执行一次 `Normalize -> Validate`
    - `Polish` 不允许回退到旧版“全量规范大提示词”，必须使用 issue_code + fix_hint 的短提示词
    - Writer 内部最大修复轮次默认 2 轮；超限后将 `ValidationReport` 透传给 Critic

#### F5: 质量保障 ✅
- **FR5.1**: 答案正确性验证
  - 反向解题：能根据题目条件推导出唯一答案
  - 答案一致性：Critic推导答案与生成答案一致
  - 计算验证：计算题自动验证计算结果

- **FR5.2**: 解析逻辑性验证
  - 解析必须与答案一致
  - 解析必须引用教材原文
  - 解析必须解释错误选项的误区
  - **多选题解析逐项覆盖率（新增）**：
    - 适用范围：题型为多选题时。
    - Critic 必须显式计算并输出：
      - `multi_option_coverage_rate: float`（0.0–1.0，小数保留两位），表示“被解析到的选项数 / 总选项数”。
      - `missing_options: list[str]`，为未被逐项解释到的选项字母列表（如 `["B", "E"]`），按字母升序排序。
    - 逐项覆盖定义：只有当解析中对每个选项（包含正确与错误）都进行了**单独点名或明确对应**（例如“选项B……”“B项属于……”等），才视为已覆盖；仅用一句“其他选项都是因为条件不满足”这一类笼统表述，不计入逐项覆盖。
    - 通过条件：多选题要求 `multi_option_coverage_rate == 1.0` 且 `missing_options` 为空；否则解析视为结构不充分。
    - 失败时的强制报错文案：
      - 若 `0 < multi_option_coverage_rate < 1.0`，Critic 的 `quality_issues` 中必须包含文案：`“多选解析逐项覆盖率不足：X.xx”`（X.xx 为格式化后的覆盖率）。
      - 若 `missing_options` 非空，则还必须包含：`“多选解析未覆盖选项：B,E”`（将缺失选项字母按升序用英文逗号拼接）。
  - **解析首段结构（三要素强约束，新增）**：
    - 解析的第一段必须同时包含以下三类信息：
      - **目标题内容**：即路由/教材路径的前三个标题（知识点层级）；解析中须有该内容，但不要写「目标题：」字样，直接写路由前三个标题即可。
      - **掌握程度分级**：显式给出掌握程度标签之一：`了解/熟悉/掌握/应用`，且优先与知识切片中的“掌握程度”字段保持一致。
      - **教材原文引用**：至少包含一句贴近教材原文表述的引用（如“教材原文指出：……”“原文规定：……”），不得删掉关键条件或限定。
    - Critic 必须输出以下结构化标志与问题列表，用于对齐离线 Judge：
      - `first_part_missing_target_title: bool`（是否缺少目标题内容，即路由前三个标题）
      - `first_part_missing_level: bool`（是否缺少掌握程度分级）
      - `first_part_missing_textbook_raw: bool`（是否缺少教材原文引用）
      - `first_part_structured_issues: list[str]`（首段结构问题列表，如“缺少目标题内容（路由前三个标题）”“缺少掌握程度分级”“缺少教材原文引用”“首段结构顺序混乱”等）
    - 通过条件：上述三个 missing 标志均为 False，且 `first_part_structured_issues` 为空；否则解析结构视为不合格，必须在 Critic 的 `quality_issues` 中以 **minor/major** 问题形式体现出来，不得仅给出模糊的“解析不清楚”描述。

- **FR5.3**: 计算结果验证 (如适用)
  - 自动提取参数
  - 调用计算器验证
  - 支持多步计算

- **FR5.3.1**: 解析重写充分性验证（教材原文 vs 模型自述，新增）
  - 适用范围：非纯计算步骤描述的文字性解析部分。
  - Critic 必须判断解析在多大程度上属于“对教材原文的直接复读”还是“模型基于原文自己讲清楚了为什么答案如此”，并输出：
    - `analysis_rewrite_sufficient: bool`
    - `analysis_rewrite_issues: list[str]`
  - 判定原则：
    - 允许保留必要的法律条文名、政策名、专有名词等**术语级短语**，但整体句式和推理链条不能是教材句子的简单复制粘贴。
    - 如出现以下任一情形，则 `analysis_rewrite_sufficient = False`，并在 `analysis_rewrite_issues` 中写明原因（示例）：
      - “大段文字与教材原文高度一致，仅替换少量数字或名词。”
      - “解析只机械复述条文条款，没有补充任何‘为什么如此’的推理说明。”
      - “解析主要依赖‘见教材原文’等指代性描述，而没有展开解释。”
  - 通过条件：`analysis_rewrite_sufficient == True`；否则该问题至少应被标记为解析层面的 **major** 质量问题，并推荐 `fix_explanation` 或 `fix_both` 修复策略，即使当前答案本身是正确的。

- **FR5.4**: 问题分类 ✅
  - **严重问题 (major)**：无法推导唯一答案、答案错误、遗漏条件、难度不符合要求
  - **轻微问题 (minor)**：解析不清晰、格式问题
  - **通过零缺陷原则**：仅当所有校验维度“问题清单”为空时，Critic 才允许返回 `passed = true`；一旦存在任意 error / warning / issue（包括 Writer / Critic / 代码级校验输出），均必须视为“不通过”，`critic_result.passed = false`。

- **FR5.5**: 智能决策机制 ✅
  - **通过** → 结束流程（满足“零缺陷通过”原则）
  - **轻微问题** → Fixer 修复 → 回到 Critic 验证
  - **严重问题** → Router 重新路由 → 重新生成
  - **超限 (retry≥3)** → 自愈机制（bypass Router直接重新生成）
  - **泄题强制重路由**：一旦第 3 层严谨性审计给出 `rigor.leakage_still_invalid = true` 且证据字段完整（题干片段 `stem_snippet` 与正确项片段 `option_snippet` 均为非空、可对齐到具体短语），在线出题链路中的 Critic 必须：
    - 将该问题视为 **严重问题 (issue_type = "major")**；
    - 强制设置 `critic_result.passed = false`；
    - 在智能决策阶段直接走 `reroute → router` 重新出题，而不是交给 Fixer 在当前题目基础上“修修补补”。

- **FR5.6**: 数据质量过滤
  - 去除 NaN 母题
  - 验证必需字段（题干、选项、答案、解析）
  - 过滤空字符串

- **FR5.7**: 题目质量硬性约束 ✅
  - **禁止模糊用语**：题干中禁止使用"实实在在的特点"、"重要的信息"、"关键因素"等模糊表述
  - **选项维度一致性**：所有选项必须在同一维度内做区分，禁止跨维度
  - **选项父子类层级冲突（疑似多解）**：
    - 定义：当单选/多选题的选项组合中同时出现同一概念的“父类/子类”或“上位/下位”关系（如“住宅”“存量住宅”“二手房”），且在教材切片知识图谱中被判定为上下位关系时，视为**疑似层级冲突**
    - 约束：此类题目默认判定为“存在多解风险，需要结合题干额外条件才能唯一确定”，Critic 节点必须输出结构化字段：
      - `option_hierarchy_conflict_flag: bool`：是否命中父子类层级冲突
      - `option_hierarchy_conflict_pairs: List[(option_a, option_b, relation)]`：触发冲突的选项对及其关系（如 parent/child）
      - `option_hierarchy_conflict_message: str`：面向审核人的提示文案，推荐口径：“单选题选项疑似层级冲突（父/子类或上下位关系），需结合题干判断是否存在多解风险”
    - 裁决：该维度本身不直接决定题目整体通过/不通过，但必须作为质量报告中的独立风险条目展示，并由人工结合题干做最终裁决
  - **禁止定义题与对工作无帮助题**：题目须对经纪人工作有帮助（可为场景化实操题，也可为理解规则/合规/文化等）。禁止：（1）直接问定义的题目（如"以下哪个条件是二套房？"）；（2）仅考「概念归类/概念辨析」、对实际作业无帮助的题目；（3）**仅考「教材把哪一条称为核心/主要/关键」的刁钻题**——实务上多选项都合理、选对只靠记教材措辞时，Critic 应判为对工作无帮助并驳回或要求改写成对工作有指导意义的考法。
  - **禁止常识与切片表述易冲突的题目**：若考点在常识理解上与切片原文容易产生偏差（如常人认为“新建”=未交易过，而教材将“再次上市交易”也归为二手房），导致考生按常识选错或觉得题目/教材没写清楚，则不应出此类题；若必须考该口径，须在题干或解析中明确限定为教材口径并说明与日常用语的区别。
  - **禁止因主体/视角歧义产生的题目（流程与步骤类）**：若切片中的流程、步骤描述**未明确每一步的执行主体或视角**（如谁来做、从谁的角度），则不出因「主体或视角不同会产生歧义」的题目或选项。例如：流程写“步骤A→步骤B→步骤C”但未区分当事人操作与部门操作，则不宜出“最后一步是？”或依赖“当事人角度的最后一步”与“流程顺序最后一步”区分的选项，否则易产生多解。**通用规则**：流程/步骤类切片若省略主体或视角，不出依赖该区分才能唯一确定的题目或选项。
  - **禁止选项与题干条件相悖**：任一选项不得与题干中已明确给出的条件、前提或设定在逻辑上矛盾；题干已设定某事实成立时，选项中不得出现与该事实相悖的表述，否则 Critic 判为不合格并驳回或要求修改选项。
  - **正确选项须完整覆盖考点关键要素**：若教材/切片对某概念、流程或规则明确了多个并列要点，正确选项不应只表述其中部分要点而遗漏其他关键要素，否则判为正确项不完整，Critic 判不合格或要求补全正确项。
  - **唯一正确性**：单选题确保只有一个选项严格符合教材原文，其他选项须有明确错误点
  - **数据重构**：严禁直接照搬原文案例中的具体人名、金额、日期、房产面积
  - **禁止题干直接给出答案**：题干中不得直接包含正确答案的关键词，导致考生无需理解即可通过文字匹配选出答案（如题干说"经纪人介绍了商业贷款"，然后问"商业贷款最核心的特征"）
  - **允许答案与教材原文一致**：正确答案选项可以与教材原文定义一致，这是正常的考察方式，不应被视为质量问题
  - **禁止AI幻觉/非人话**：禁止出现不符合中国房地产业务习惯的生造词（如"外接"代替"买方/受让方"、"上交"代替"缴纳"）
  - **难度差异化要求**：低难度题不强制干扰项“优秀程度”，高难度题必须满足高质量干扰项要求

- **FR5.8**: 地理一致性约束 ✅
  - **地理继承**：如果教材明确限定了城市（如"北京市"），题干场景必须设定在该城市（或其下辖区县）
  - **严禁无关城市**：绝对禁止出现原文未提及的其他具体城市名（如上海、深圳、广州等）
  - **母题城市替换**：即使参考母题中写的是其他城市，必须自动替换为原文指定的城市或通用化
  - **干扰项特例**：干扰项中允许出现其他城市作为错误选项，但题干场景和正确答案必须基于教材指定城市
  - **通用规则处理**：如果原文是通用规则（未提及特定城市），题干不得写具体城市，可用"某市"或不提及地点

- **FR5.9**: 时间逻辑约束 ✅
  - **时间继承**：如果原文未给出具体时间，题干与解析不得添加具体年份/日期；仅保留相对时间（如"满5年"）
  - **时间使用规则**：若原文明示时间，才可使用对应年份/日期；判定逻辑必须严格遵循教材规则（如"满5年"的计算）
  - **时间干扰项**：允许设计关于时间的干扰项（如设置一个时间未满的情景作为错误选项），但解析必须清晰指出不符合哪条时间规则
  - **计算时间精度**：时间/日期题必须用 datetime 精确到天，禁止用年份直接相减

#### F6: 反馈循环机制 ✅
- **FR6.1**: Fixer → Critic 循环
  - 修复后自动回到验证
  - 最多3次修复循环

- **FR6.2**: Critic → Router 重路由
  - 严重问题触发重新路由
  - Router检测重路由并清理旧状态

- **FR6.3**: 状态清理
  - Router检测重路由并清理旧状态
  - 保留必要的上下文（prev_final_json, prev_critic_feedback）

- **FR6.4**: retry_count 管理

  - 超限后触发自愈机制（bypass Router直接重新生成）
  - 自愈机制：当retry_count >= 3时，直接调用Fixer重新生成，不再经过Router

- **FR6.5**: 智能决策机制 ✅

### 3.6 成本与单位经济可观测需求（商业 ROI 与数据飞轮）

- **NFR6.1**: CPVQ（单题有效生产成本）可观测
  - **定义**：CPVQ = 当次任务总成本 (total_cost) / 有效题目数 (saved_count)。有效题目指 Critic 通过且成功入库的题目。
  - **落库**：每次 QA Run 落盘时，`batch_metrics` 中必须包含 `cpvq`；当 `saved_count > 0` 时为数值（total_cost / saved_count），否则为 null 或明确标注无有效题目。
  - **与现有指标区分**：`avg_cost_per_question` 为总成本/生成题数（毛成本）；CPVQ 仅按有效产出算成本，用于商业算账。

- **NFR6.2**: CPVQ 告警与阈值
  - **阈值配置**：QA 阈值配置支持可选字段 `cpvq_max`（单题有效生产成本上限）。
  - **告警规则**：当 `saved_count > 0` 且 `cpvq > cpvq_max` 时，产生 batch_metric 类告警并写入 `qa_alerts.jsonl`。
  - **可选**：当 `saved_count == 0` 且 `generated_count > 0` 时，可产生「本批无有效题目」类告警，便于运营关注。

- **NFR6.3**: 单位经济视图可导出/可查
  - **成本侧**：已有 `cost_summary`（by_question / by_node / by_model）及 per-question tokens、cost_estimate，需在 Run 详情或导出中稳定呈现，形成「单题成本 + 按节点/模型拆分」的单位经济视图。
  - **收益侧**：短期以 `saved_count`、`hard_pass_rate` 等作为有效产出指标；后续若线上有题目使用数据，可扩展为「成本 × 使用量」的 ROI 分析。

- **NFR6.4**: 管理端展示 CPVQ
  - 在管理端「任务详情」或「QA Run 详情」中，与 `avg_cost_per_question`、`saved_count`、`hard_pass_rate` 并列展示 CPVQ，便于对比毛成本与有效成本。

- **NFR6.5**: 数据飞轮与线上线下打通（目标态）
  - **数据飞轮**：利用 `qa_runs.jsonl`、`gen_tasks.jsonl` 做失败原因与成本聚合，反哺提示词、模型选择与重试策略，持续优化 CPVQ 与通过率。
  - **线上线下打通**：题目在「生成 → 入库 → 考试」全链路使用统一 question_id；线下成本视图与线上使用数据（若存在）可关联，支持「单题 ROI」或「高成本低使用题目」分析。具体数据格式与对接方式由后续迭代定义。

    - **附录公式表拆分与归属**：若检测到“附录 计算公式汇总表”的表格，需拆分为独立切片；并依据“公式等号左侧名词”将公式归入最匹配的知识切片（含“附录”字样的切片需重点匹配）
    - **路由最多 5 级**：构建路径时只取 `path_stack[1:6]`（即层级 1～5）。若存在第 6 级标题（如（1）、1.），不作为路径的一部分，视为「5 级目录下的内容」归入当前 5 级切片；路径最大深度为 5 级。
    - **5 级目录下的内容通过 embedding 再切片**：对同一 5 级路径下的正文/表格等按段落或块划分，使用 BGE（`BAAI/bge-small-zh-v1.5`）计算向量；按相似度或固定策略（如块边界、语义聚类）划分子切片，产出多个原子切片时路径均为该 5 级路径（可带可选子标题元数据）。
    - **内容不长且关联度高则合并为一个切片**：同一 5 级路径下，若子块总长度低于设定阈值且块间 BGE 相似度高于设定阈值，则合并为一个切片输出，不强制拆成多切片。
    - **短切片回退上一级合并**：切片完成后若某切片内容（context_before + context_after + tables + examples + formulas 等）字符数小于设定阈值（默认 200 字），则将该路径的**上一级**下全部切片作废，在上一级维度重新合并为单一切片；新切片以「上一级路径」为完整路径、以合并后的全部内容为结构化内容，保证单切片长度不低于阈值；可配置 `MIN_SLICE_CONTENT_CHARS`，并循环执行直到无短切片或已到根路径。
  - **格式**: JSON Lines，每行一个知识点（`json.loads(line)`）
  - **必需字段**:
    - `完整路径`: 字符串，知识点层级路径（如"第一篇 > 第一章 > 第一节 > ..."）
    - `核心内容`: 字符串，知识点核心文本内容
      - 如果缺失，从`结构化内容`自动构建（`context_before` + `context_after` + `tables` + `formulas` + `examples`）
  - **可选字段**:
    - `掌握程度`: 字符串，如"了解"/"熟悉"/"掌握"或"未知"
    - `结构化内容`: 字典，包含：
      - `context_before`: 上下文前置文本
      - `context_after`: 上下文后置文本
      - `tables`: 核心表格数据列表
      - `formulas`: 核心公式列表
      - `examples`: 教材原题列表（知识点切片内置的题目示例，最高优先级参考）
      - `key_params`: 关键参数列表（可选）
    - `Bot专用切片`: 字符串，格式化后的文本（可选）

- **母题库**: `存量房买卖母卷ABCD.xls` (408 道) ✅
  - **文件路径**: `HISTORY_PATH = "存量房买卖母卷ABCD.xls"`

## 5. 约束条件

### 5.1 技术约束
- **C1**: 必须使用 Streamlit 作为 UI 框架 ✅

- **C2**: 必须使用 LangGraph 进行智能体编排 ✅

- **C3**: LLM 需支持 JSON 格式输出 ✅
  - 所有节点都要求LLM返回JSON格式
  - 支持从markdown代码块中提取JSON

- **C4**: Python 环境 ✅
  - 需要安装相关依赖包（pandas、openpyxl、pydantic、scikit-learn、openai、streamlit、langgraph、langchain等）

### 5.2 业务约束
- **C5**: 题目内容必须 100% 准确，不得出现幻觉
- **C6**: 题干中禁止出现"根据材料"、"依据参考资料"等提示语
- **C7**: 干扰项必须似是而非，不能一眼假
- **C8**: 必须出场景化案例题，禁止定义题
- **C9**: 选项维度必须一致，禁止跨维度
- **C10**: 严禁直接照搬原文案例中的具体数据（必须做数据重构）
- **C11**: 难度值必须符合用户指定的难度范围
- **C12**: 题型必须符合用户选择（单选/多选/判断）
- **C13**: 出题筛选条件必须符合用户选择（基础概念/理解记忆、实战应用/推演、随机）
- **C14**: 地理一致性：教材限定城市时，题干必须在该城市；通用规则不得写具体城市
- **C15**: 时间逻辑：原文未给具体时间时，不得添加具体年份/日期；仅保留相对时间
- **C16**: 禁止题干直接给出答案：题干中不得直接包含正确答案的关键词，导致考生无需理解即可通过文字匹配选出答案。但允许正确答案选项与教材原文定义一致。
- **C16-b**: 第三层泄题判定（rigor.leakage_still_invalid）：
  - **适用范围**：仅用于第 3 层严谨性审计节点（rigor critic），字段为 `rigor.leakage_still_invalid`。
  - **泄题成立条件（需同时满足）**：
    1. **关键表述高度同义**：题干中的关键术语 / 情景短语 / 数值组合，与正确选项中的关键表述高度一致或同义；
    2. **几乎无需理解教材即可唯一锁定**：考生几乎不需要理解教材内容或业务规则，只要抓住这句关键表述，就能在选项中唯一锁定正确选项。
  - **禁止的判定方式**：
    - 禁止仅凭关键词重合判定泄题：
      - 如果题干和正确项只是共享通用术语 / 主题词（如“贷款、利率、税费、合同、客户、经纪人、门店、服务”等），不得直接认为泄题；
      - 如果题干中涉及的概念在多个选项里都以类似方式出现，仍然需要理解/推理才能得出答案，则不得视为泄题。
  - **证据要求（强约束）**：
    - 当 `rigor.leakage_still_invalid = true` 时，必须在 `rigor.issues` 中提供两段证据：
      1. 题干中导致泄题的关键表述原文片段；
      2. 正确选项中与之相同或同义的关键表述片段。
    - 若上述任一片段缺失或无法唯一指向具体文本，提示词必须要求将 `rigor.leakage_still_invalid` 改判为 `false`。
  - **题型语义说明**：
    - 基础概念/理解记忆题：侧重描述为“题干中的关键概念术语 / 定义短语 与正选中的关键表述高度一致或同义”；
    - 实战应用/推演题：侧重描述为“题干中的关键业务情景 / 术语 / 数值组合 与正确选项关键表述高度一致或同义”。
- **C17**: 禁止生造词：必须使用标准业务术语，禁止AI幻觉词汇
- **C18**: 禁止"最XX"考法：禁止用"最重要/最关键/重点/主要"等表述
- **C19**: 唯一答案强制校验：逐条假设每个错误选项为真，验证是否"必错"

### 5.3 数据约束
- **C20**: 知识库和母题库为静态数据，不可修改 ✅
  - 知识库和母题库文件为只读，系统仅读取，不写入

- **C24**: 母题库字段约束 ✅
  - **必需字段**（用于数据质量过滤）：`题干`、`选项1`、`选项2`、`正确答案`、`解析`
  - **可选字段**：`选项3-8`、`考点`、`难度值`
  - **题型识别**：
    - 判断题：选项1为"正确"且选项2为"错误"
    - 多选题：正确答案长度>1且所有字符在A-E中

- **C25**: 输出题目字段约束 ✅
  - **必需字段**（`ExamQuestion`模型）：`题干`、`选项1`、`选项2`、`正确答案`、`解析`、`难度值`
  - **可选字段**：`选项3-8`（默认空字符串）
  - **正确答案格式**：正则表达式`^[ABCDEFGH]+$`（A-H或组合）
  - **难度值范围**：0.0-1.0（`ge=0, le=1`）
  - **自动生成字段**：`考点`、`一级知识点`、`二级知识点`、`三级知识点`、`四级知识点`、`来源路径`、`_was_fixed`、`是否修复`

## 9. 风险与应对

### 风险 1: LLM 生成不稳定
- **应对**: Critic 验证 + Fixer 修复 + 最多 3 次重试 + 自愈机制

### 风险 2: 母题数量不足
- **应对**: 仅使用知识点映射文件；无映射则不返回母题

### 风险 3: 计算器调用失败
- **应对**: 错误捕获 + 日志记录 + 降级为概念题

### 风险 4: 数据质量问题
- **应对**: 数据验证 + NaN 过滤 + 必需字段检查

### 风险 5: 难度值不符合要求
- **应对**: ✅ 多节点验证（Writer、Critic、Fixer）+ 自动调整机制

### 风险 6: 题型不符合要求
- **应对**: ✅ 所有节点严格遵循题型约束 + 修复模式也遵循

### 风险 7: 模式不符合要求
- **应对**: ✅ 提示词中明确区分三档筛选条件 + 修复模式也遵循

### 风险 8: 代码执行安全问题
- **应对**: ✅ 沙箱执行环境 + 模块白名单 + 执行时间参数（待补强超时中断） + 内置函数限制

### 风险 9: 地理/时间逻辑错误
- **应对**: ✅ Writer节点检查 + Critic节点验证 + 明确Fail条件

### 风险 10: 同义反复/泄题风险
- **风险说明**: 题干中的关键表述与正确选项出现高度文本重合或同义复述，可能导致考生无需理解教材内容，仅凭表面词句即可锁定正确答案，形成“泄题”。
- **应对机制**:
  - **Phase 1 泄题弱信号（DeterministicFilter._check_leakage）**：
    - 仅基于题干与正确选项之间的高区分度关键词重合给出弱信号；
    - 示例输出：`题干疑似泄题：出现与正确选项高度重合关键词`；
    - 只作为 warning，不直接判题泄题，也不触发 reject。
  - **第 3 层严谨性审计（rigor.leakage_still_invalid）**：
    - 最终泄题判定由第 3 层 rigor 节点完成，严格遵守 C16-b 中的双条件 + 双证据规则；
    - 仅当“关键表述同义 + 几乎无需理解教材即可唯一锁定”同时满足时，才允许 `rigor.leakage_still_invalid = true`；
    - 必须给出“题干片段 + 正确项片段”两段证据，否则强制改判为 `false`；
    - 明确禁止仅凭关键词重合或通用业务术语重合直接判泄题。

### 风险 11: AI幻觉/生造词
- **应对**: ✅ Critic节点检测生造词（如"外接"、"上交"等非标准术语）
