# 权限模型与角色矩阵

本文档说明当前管理后台的鉴权模式、角色定义、租户隔离规则，以及各角色的大致能力边界。

## 1. 鉴权模式

当前系统支持两种模式：

- `legacy`
  - 默认模式
  - 通过请求头 `X-System-User` 标识当前系统号
  - 角色、租户和权限来自本地 ACL 文件

- `oidc`
  - 通过 `Authorization: Bearer <token>` 鉴权
  - 优先使用 token 中的 `role`、`tenants`、`permissions`
  - 若 token 里没有租户信息，会回退到本地 ACL 读取该系统号的租户绑定

## 2. 角色定义

当前角色共 4 类：

- `platform_admin`
  - 平台管理员
  - 可访问所有城市
  - 可管理城市、系统号、全局 Key

- `city_admin`
  - 城市管理员
  - 只访问绑定城市
  - 具备本城市完整业务操作权限

- `city_teacher`
  - 教研/教师角色
  - 只访问绑定城市
  - 可做教材上传、切片审核、映射确认、查看出题结果
  - 不具备执行出题、改题库、发布版本等写操作

- `city_viewer`
  - 只读角色
  - 只访问绑定城市
  - 只能查看，不可执行审核、出题、发布等写操作

## 3. 租户隔离规则

系统按 `tenant_id` 做城市级隔离。

核心规则：

- 用户只能访问自己绑定的城市
- 即使 URL 中手工传入其他 `tenant_id`，只要不在可访问租户列表内，就会被拒绝
- `platform_admin` 在本地 ACL 模式下自动拥有全部城市
- `platform_admin` 在 OIDC 模式下，如果 token 没有显式传租户，也会回退为全部默认城市或本地可见城市

## 7. OIDC 模式下的特殊规则

OIDC 模式和 legacy 模式最大的差异在于权限来源：

- 若 token 中带 `permissions`，优先信任 token
- 若 token 中权限为空，才可能通过角色兜底
- 若 token 中既没有目标权限，又不是 `platform_admin`，后端会直接拒绝
- 若 token 中没有租户，系统会尝试回退到本地 ACL

因此：

- OIDC 接入时，建议在 IdP 中显式下发 `role`、`tenants`、`permissions`
- 不建议只传一个用户名而不传权限

## 8. 常见拒绝原因

常见错误码与含义：

- `UNKNOWN_USER`
  - 本地 ACL 中找不到该系统号

- `TENANT_FORBIDDEN`
  - 当前用户未绑定该城市

- `PERMISSION_DENIED`
  - 用户已进入该城市，但没有对应权限码

- `PERMISSION_MISSING_IN_TOKEN`
  - OIDC token 没带权限，且后端无法据此放行

- `PLATFORM_ADMIN_REQUIRED`
  - 当前接口只允许平台管理员访问

## 9. 建议运维做法

- 平台管理类操作尽量统一使用 `admin` 或明确的平台管理员账号
- 业务同学日常使用尽量分配为 `city_admin` 或 `city_teacher`
- OIDC 正式接入后，建议将角色与权限收口到统一 IdP，不要长期依赖本地 ACL 兜底
