# 版本发布流程

本文档说明版本管理页面的发布前置条件、发布动作、Git 提交行为，以及失败时的排查思路。

## 1. 发布对象

当前“版本发布”不是重新生成题目，而是把一组已经完成质量评估的 `run` 标记为一个人工发布版本。

发布记录至少包含：

- `version`
- `release_notes`
- `run_id`
- `run_ids`
- `published_at`
- `published_by`

## 2. 发布前置条件

后端对每个被选中的 `run` 都会校验以下条件：

### 2.1 `version` 必填

失败示例：

- `version 必填`

### 2.2 `run_ids` 至少 1 个

失败示例：

- `run_ids 必填（至少 1 个）`

### 2.3 run 必须真实存在

失败示例：

- `该 run_id 不存在: xxx`

### 2.4 run 必须有落库题

判断条件：

- `saved_count > 0`

失败示例：

- `发布版本前所选 run 须有落库题目（saved_count>0）`

### 2.5 run 必须已经跑过离线 Judge

判断条件满足任一即可：

- `batch_metrics.judge_pass_rate` 存在
- `batch_metrics.judge_pass_count` 存在
- `questions[*].offline_judge` 中至少有一题存在结果

失败示例：

- `发布版本前所选 run 的落库题目须已跑过离线 Judge`

## 3. 页面中的“可发布 run”

版本管理页不会展示所有 run，而是优先展示 `release_eligible === true` 的 run。

该标记的判定规则与后端一致：

- `saved_count > 0`
- 且 `has_judge == true`

如果不满足，页面会保留原因字段：

- `无落库题目，请先跑出题任务并确保有题目入库`
- `落库题目未跑离线 Judge，请对该 run 执行「运行 Judge」`

## 4. 标准发布流程

建议按以下顺序操作：

1. 先完成一次出题任务，并确认 `saved_count > 0`
2. 对该 run 执行离线 Judge
3. 在质量评估页确认关键指标可接受
4. 打开版本管理页，选择可发布 run
5. 填写 `version`
6. 填写 `release_notes`
7. 视情况决定是否勾选“同时提交 Git”
8. 点击“发布版本”

## 5. 多 run 发布

接口支持一次发布绑定多个 `run_ids`。

这通常用于：

- 多个任务共同构成一个版本基线
- 同一个发布版本希望覆盖多个 run

注意：

- `run_ids` 会去重
- 发布记录里仍会保留主 `run_id`，它取 `run_ids[0]`

## 6. Git 提交行为

如果勾选“同时提交 Git”，发布接口会额外触发一次真实 Git 提交与推送。

可传参数包括：

- `git_repo_url`
- `git_user_email`
- `git_user_name`
- `git_username`
- `git_token`
- `git_commit_message`
- `git_push_branch`

前端默认值：

- 仓库地址默认指向内部仓库
- `git_commit_message` 默认是 `[紧急]fix`
- 推送分支默认是 `main`

- [admin-web/src/pages/VersionManagementPage.jsx](/Users/panting/Desktop/搏学考试/AI出题/admin-web/src/pages/VersionManagementPage.jsx#L28)

## 7. Git 提交范围

发布触发的 Git 提交不是“全仓提交”，而是只提交代码类文件。

会被纳入的典型后缀包括：

- `.py`
- `.js/.jsx/.ts/.tsx`
- `.css/.scss/.less`
- `.html`
- `.json/.yml/.yaml/.toml/.ini/.cfg`
- `.sql`
- `.sh/.bash/.zsh/.command`
- `.txt`

不会纳入的典型路径包括：

- `data/`
- `admin-web/node_modules/`
- `admin-web/dist/`
- `.venv/`
- `__pycache__/`
- `tmp/`

结论：

- 运行时数据、构建产物、虚拟环境不会随发布自动推送

## 8. Git 推送回退逻辑

若目标分支 push 失败，且失败原因是非 fast-forward 或被拒绝，系统会尝试回退到自动分支：

- `auto-release/{tenant_id}/{timestamp}`

这意味着：

- 发布记录仍然会创建
- Git 结果里会带 `warning`
- 代码并不是推到原目标分支，而是推到了回退分支

## 9. 接口返回

发布成功时，返回体主要包含：

- `release`
- `suggested_commit_message`
- 可选的 `git`

其中 `git` 里常见字段有：

- `ok`
- `message`
- `error`
- `warning`
- `commit_message`
- `remote_url`
- `push_branch`
- `requested_push_branch`

## 10. 常见失败场景

### 10.1 run 不可发布

先检查：

- 是否真的有 `saved_count > 0`
- 是否已经执行过离线 Judge

### 10.2 Git 提交失败

常见原因：

- 当前目录不是 Git 仓库
- `git add` 失败
- `git commit` 失败
- 远端仓库地址不可达
- 凭证不正确
- 目标分支无权推送

### 10.3 页面里看不到某个 run

先看：

- run 是否在最近查询范围内
- `release_eligible` 是否为 `false`
- `release_eligible_reason` 写的是什么

## 11. 常见误区

- “发布版本”不会重新跑出题，也不会自动跑 Judge
- 只生成了题但没落库，不可发布
- 落库了但没跑离线 Judge，也不可发布
- 勾选 Git 后会真实 push，不是模拟按钮
- Git 失败不一定阻止发布记录创建，要同时检查 `release` 和 `git` 结果
