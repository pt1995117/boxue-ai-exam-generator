# 映射审核手工补全规则

本文档说明映射审核页面中“手工补全母题”的适用场景、字段要求，以及为什么有些映射无法直接点通过。

## 1. 背景

映射审核的目标不是只把切片和某个母题粗暴连起来，而是保证这条映射背后的母题信息足够完整，可被后续出题流程稳定使用。

因此，审核通过前，系统会校验母题是否至少具备：

- `题干`
- `选项`
- `解析`

若缺其中任意一项，后端会拒绝 `approved`。

## 2. 审核接口

批量审核接口：

- `POST /api/{tenant}/mappings/review/batch`

常见入参：

- `map_keys`
- `confirm_status`
- `comment`
- `reviewer`
- `material_version_id`
- `target_mother_question_id`
- `manual_question_stem`
- `manual_question_options`
- `manual_question_explanation`

## 3. `map_key` 含义

`map_key` 通常形如：

- `slice_id:question_index`

它表示：

- 左侧是切片
- 右侧是候选母题索引

如果 `map_key` 格式本身都不合法，后端会直接认为该条映射缺少 `题干/选项/解析`，无法通过。

## 4. 什么时候需要手工补全

以下场景通常需要手工补：

- 候选母题本身没有完整题干
- 只有题干，没有选项
- 有题干和选项，但没有解析
- 想改绑到另一个母题 ID
- 历史母题记录不完整，但业务上又确认这条映射应通过

## 5. 后端判定优先级

当 `confirm_status == approved` 时，后端按以下顺序判断“母题是否完整”：

### 5.1 先看手工补全内容

如果你传了：

- `manual_question_stem`
- `manual_question_options`
- `manual_question_explanation`

且这三者已经构成完整母题，则以后端手工内容为准。

### 5.2 否则看目标母题

若手工内容还不完整，后端会取：

- `target_mother_question_id`
- 若未显式指定，则退回 `map_key` 自带的原候选母题 ID

然后去历史母题记录里取该母题详情做完整性校验。

结论：

- 手工补全优先级高于历史母题
- 但如果手工只补了一半，系统不会自动“半手工半通过”，仍然会继续检查目标母题

## 6. “完整母题”的判定规则

后端只看三项：

- `题干` 非空
- `选项` 至少 1 个非空项
- `解析` 非空

当前不会因为“没填正确答案”而阻止通过。

但这不代表正确答案不重要，只是映射审核的硬门槛目前没有把它列入阻断条件。

## 7. `manual_question_options` 传参规则

后端兼容两种传法：

- 字符串
  - 按换行拆分成多条选项
- 数组
  - 数组中每个非空字符串视为一个选项

如果最终拆出来没有任何有效选项，仍会被视为“缺少选项”。

## 8. 常见失败返回

当审批通过失败时，后端会返回：

- `MAPPING_NOT_READY`

消息形态类似：

- `以下映射未补全母题题干/选项/解析，不能通过审核：...`

并会列出样例：

- 哪个 `map_key`
- 缺的是 `题干/选项/解析` 中哪些字段

## 9. 前端操作建议

建议审核同学按以下顺序处理：

1. 先确认当前候选母题是否就是目标母题
2. 如果不是，先填写 `target_mother_question_id`
3. 查看目标母题是否已经有完整题干、选项、解析
4. 若不完整，再补 `manual_question_stem`
5. 补 `manual_question_options`
6. 补 `manual_question_explanation`
7. 最后再点通过

## 10. 适合驳回而不是补全的情况

以下情况更适合直接驳回：

- 这条映射本身就是错配
- 当前切片和任何候选母题都不相关
- 即使手工补字段，也无法证明这条母题真的对应当前切片

说明：

- 手工补全是“修正文档完整性”，不是“强行制造一条本来不存在的映射”

## 11. 常见误区

- 指定了 `target_mother_question_id` 不代表一定能通过，还要看目标母题是否完整
- 只补题干不补解析，仍然不能通过
- 只补解析不补选项，仍然不能通过
- 映射审核的硬门槛当前不检查正确答案，但后续出题质量仍会受其影响
